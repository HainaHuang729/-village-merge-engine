# GAME_PILLARS

Game Pillars 是项目的不可妥协支柱。未来任何功能都必须至少服务其中一个支柱，最好服务多个。

## Pillar 1：村庄每天都要更活一点

村庄不是奖励界面，也不是背景。

每个完成的建设都必须带来可感知的生活变化：

- 新的人出现。
- 新的声音出现。
- 新的路径被使用。
- 新的日常行为发生。
- 新的村民对话或小事件出现。

错误做法：

- 鸡舍完成后只显示 `鸡蛋产量 +20%`。
- 建筑完成后只是图标点亮。
- 村庄变化只存在于任务文案里。

正确方向：

- 鸡舍完成后，村里真的出现鸡。
- 老人开始喂鸡。
- 早晨有鸡叫。
- 孩子偶尔追鸡。
- 之后某天鸡跑到村口，形成一个轻事件。

## Pillar 2：建设爽优先于合成爽

Merge 的爽感是过程，建设的爽感是结果。

玩家最强的奖励瞬间不应该是两个物品合成成功，而应该是：

```text
最终产物飞入村庄，村庄发生永久变化。
```

设计优先级：

1. 建设爽。
2. 拆货爽。
3. Merge 爽。
4. 整理爽。
5. 数值反馈。

如果某个功能让玩家长时间停留在棋盘里，却没有推动村庄变化，它需要被压缩。

## Pillar 3：每天十分钟，必须有一个完整情绪闭环

微信小游戏的优势不是长时间沉浸，而是高频轻触达。

一次日常游玩应该包括：

- 回到村口。
- 看到今天的新情况。
- 处理货车。
- 完成若干 Merge。
- 推进一个建设或事件。
- 看到村庄变化或生活反馈。
- 带着“明天再来看”的期待离开。

十分钟内不能只让玩家做材料准备。

## Pillar 4：玩家是返乡大学生，不是全能管理者

玩家身份决定游戏气质。

玩家不是用权力改造村庄，而是用行动连接资源：

- 帮老人解决小问题。
- 把城市物流带回村里。
- 用互联网帮村里卖东西。
- 组织小活动。
- 吸引年轻人返乡。
- 让游客愿意来。

这能避免游戏滑向传统经营模拟。

## Pillar 5：永久留下的是生活，不是库存

今天消耗的是货物。

永久留下的是：

- 建筑。
- 人物关系。
- 村庄习惯。
- 每日事件池。
- 视觉和声音变化。

库存、金币、订单都只是辅助。

如果长期成长主要体现在资源数字上，项目会失去特色。

## Pillar 6：复杂度沉淀到村庄，不堆在棋盘

棋盘应该保持简单。

村庄可以越来越丰富，但棋盘不应变成满屏生产器、冷却器、碎片和锁格。

Merge 棋盘承担：

- 拆货后的整理。
- 合成短目标。
- 产生建设材料。

村庄承担：

- 规则变化。
- 事件变化。
- 人物变化。
- 长期记忆。

## Pillar 7：每个系统都要制造“今天发生了什么”

任何系统都应该能转化成村庄生活。

好的系统问题：

- 它今天会让村里发生什么？
- 它会让哪个角色有新行为？
- 它会让哪个地点变热闹？
- 它会让玩家明天期待什么？

坏的系统问题：

- 它每小时产出多少？
- 它能否提高付费转化？
- 它能否增加在线时长？

商业目标重要，但不能先于项目灵魂。
