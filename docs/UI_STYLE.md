# UI_STYLE

状态：Game 层 UI 草案。UI 只能服务 10 分钟日常体验和村庄变化表达，不应引入与 North Star 冲突的复杂任务或数值压力。

## UI 目标

UI 应服务短局操作：

- 一眼看到货车有多少货。
- 一眼看到当前目标。
- 一眼知道棋盘能不能继续拆货。
- 建筑完成和新规则出现要清晰。

## 信息优先级

第一层：

- 货车剩余货物。
- 棋盘。
- 当前任务或建筑目标。

第二层：

- 村庄入口。
- 建筑状态。
- 设置。

第三层：

- 图鉴。
- 统计。
- 活动。

MVP 只做第一层和必要的第二层。

## 主界面结构

推荐竖屏布局：

```text
顶部：货车状态 / 当前目标
中部：6x6 棋盘
底部：拆货按钮 / 村庄入口 / 简单工具
```

村庄界面：

```text
顶部：返回棋盘 / 当前村庄目标
中部：村庄地图
底部：选中建筑信息
```

## 交互原则

- 拖动是主要操作。
- 点击用于查看或交付。
- 长按用于删除或回收等危险操作。
- 可合并物品要有视觉提示。
- 不合法操作要快速回弹并给轻反馈。

## 文案原则

文案短，尽量贴合世界观。

推荐：

- “货车已满”
- “拆 1 件”
- “送去村里”
- “鸡舍完成”
- “还差 1 只母鸡”

避免：

- “体力不足”
- “产出倍率 +100%”
- “系统功能未开放”
- “请完成前置条件后解锁该模块”

## 货车 UI

需要显示：

- 当前货量。
- 最大货位。
- 下一件配送倒计时。
- 拆货按钮。

示例：

```text
货车 18/20
下一件 01:24
[拆货]
```

当棋盘满时：

```text
棋盘满了，先整理一下
```

## 棋盘 UI

要求：

- 每个格子尺寸稳定。
- 拖动时不挤压布局。
- 物品图标不会超出格子。
- 高亮不会遮挡识别。

状态：

- 空格。
- 普通物品。
- 可合并目标。
- 可交付最终物品。
- 选中物品。

## 村庄 UI

建筑状态表达：

- 未解锁：隐藏或淡轮廓。
- 可建造：显示需求气泡。
- 已完成：正常显示并可点击查看效果。

建筑信息面板：

- 建筑名。
- 当前状态。
- 还需要什么。
- 完成后解锁什么。

## 反馈原则

正反馈：

- Merge 时缩放和光效。
- 高级物品出现时更强反馈。
- 建筑完成时村庄镜头轻微聚焦。

负反馈：

- 禁止拆货只给简短提示。
- 删除物品需要明确确认。
- 不要用强烈失败音惩罚玩家。

## 微信小游戏约束

- 所有按钮要适合触屏。
- 避开系统安全区域。
- 加载时间要短。
- UI 资源尺寸控制。
- 文案不要依赖很小字体。
- 不使用复杂 DOM UI，优先 Canvas 内 UI 或轻量平台适配。
