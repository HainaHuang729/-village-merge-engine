# DESIGN_PRINCIPLES

## 1. 先设计生活，再设计系统

不要先问：

```text
这个建筑产出什么？
```

先问：

```text
这个建筑让村里多了什么生活？
```

例如鸡舍：

错误起点：

- 鸡舍每小时产鸡蛋。

正确起点：

- 村里重新有鸡叫。
- 老人多了日常行为。
- 孩子多了玩耍事件。
- 后续集市可以卖土鸡蛋。

产出可以存在，但不能是设计核心。

## 2. 所有建设都必须可感知

建成一个东西，玩家必须马上看到变化。

变化可以很小，但必须具体：

- 一个角色换了位置。
- 一个地块不再荒废。
- 一个声音出现。
- 一段对话改变。
- 一个新事件进入日常池。

不要把建设奖励藏在面板里。

## 3. 每日体验必须有结尾

微信小游戏不是让玩家无限沉迷。

每天 10 分钟应该有一个自然结尾：

- 今天的货处理得差不多。
- 今天的建设推进了。
- 今天村里有一个小变化。
- 明天有一个轻期待。

如果游戏让玩家无限刷，短期数据可能好，但长期会疲劳。

## 4. Merge 是整理，不是全部

Merge 的意义是把“混乱的货物”整理成“能改变村庄的东西”。

如果 Merge 变成纯物品链收集，它会和竞品正面撞车。

因此：

- 棋盘要简单。
- 物品链要服务建设。
- 合成终点要离开棋盘进入村庄。
- 不要让棋盘成为永久负担。

## 5. 货车是外部世界进入村庄的入口

货车不是体力条。

货车代表：

- 城市物流。
- 外部资源。
- 互联网订单。
- 返乡青年连接外部世界的能力。

设计上要避免“货车 = 每两分钟给一次点击次数”的廉价感。

货车可以带来：

- 特殊配送。
- 村民托运。
- 游客寄来的东西。
- 活动物资。
- 外部世界新闻。

## 6. 村民不是功能按钮

村民不能只负责发布任务。

村民应该有：

- 日常行为。
- 记忆。
- 关系变化。
- 对建筑变化的反应。
- 对玩家行动的温和反馈。

如果村民只是订单 NPC，游戏会变成普通经营任务链。

## 7. 不要用焦虑制造留存

不依赖：

- 过期奖励。
- 惩罚性体力。
- 不上线损失。
- 大量红点。
- 长列表任务。

Village Merge 的留存应该来自：

```text
我想看看今天村里怎么样了。
```

这比焦虑更难做，但也是项目差异化所在。

## 8. 内容更新优先更新“村庄状态”

长期更新不要只加新链和新建筑。

更应该更新：

- 村庄事件。
- 人物关系。
- 季节变化。
- 集市主题。
- 游客反馈。
- 返乡青年故事。

新物品链如果不能带来新的村庄生活，就只是内容膨胀。

## 9. 任何复杂设计都要能被一句生活语言解释

好的：

```text
村里修好了鸡舍，所以早上能听到鸡叫了。
```

坏的：

```text
鸡舍激活后，禽类标签物品的二阶产出效率提升 15%。
```

后者可以是内部实现，但不能成为玩家体验。

## 10. 当系统和情绪冲突时，优先情绪

如果一个数值设计合理，但破坏村庄情绪，应调整数值设计。

例子：

- 为了留存强迫玩家每小时上线收货，会破坏治愈感。
- 为了商业化让建筑变成产币机器，会破坏建设感。
- 为了内容消耗增加复杂订单，会破坏十分钟轻松体验。

项目不能为了短期指标牺牲核心体验。
