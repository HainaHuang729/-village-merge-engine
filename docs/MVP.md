# MVP

MVP 目标不是做完整游戏，也不是验证经济系统。

MVP 只验证一个问题：

```text
玩家完成鸡舍后，是否因为村里出现了新的生活而感到满足，并愿意明天回来。
```

如果这个问题不成立，不扩展第二条链、不加建筑、不做经济。

## 当前前置条件

Village Merge MVP 必须建立在 Reusable Merge Engine 的核心交互稳定之后。

当前状态：

- Engine 核心 Board / Move / Swap / Merge 已可测试。
- Game 层已实现第一版鸡舍闭环。
- 当前仍需手玩 Review 和微信真机适配 Review。
- Game 层已有 TruckSystem、VillageState 和内容配置一致性测试。

Engine 先完成：

- Board placement。
- Move / Drop。
- MergeRule JSON loader。
- 普通二合一。
- 基础 Save snapshot。
- 基础测试。

Game 再做 Village Merge 鸡舍闭环。

## MVP 成功标准

玩家可以：

1. 打开游戏。
2. 看到今天村口和村庄有一个轻微状态。
3. 看到货车有货。
4. 拆货获得物品。
5. 在 6x6 棋盘上移动和 Merge。
6. 合成最终物品。
7. 把鸡舍交付到村庄。
8. 看到空地变成鸡舍。
9. 看到至少 3 个生活反馈中的一部分：鸡出现、老人喂鸡、孩子追鸡、早晨鸡叫、村民短句反馈。
10. 关闭游戏时知道明天村里可能还有新变化。

如果以上闭环不成立，不扩展内容。

## 必须完成

### 技术

- 使用 `packages/merge-engine` 提供通用能力。
- `games/village-merge` 不把具体玩法写入 Engine。
- Web Demo 可运行。
- 微信小游戏适配可以后置到 Engine 核心验证后。
- 基础资源加载。
- 本地存档。
- 简单调试入口。

### 玩法

- 6x6 棋盘。
- 拖动物品。
- 空格移动。
- 相同物品 Merge。
- 1 条 Merge 链：鸡蛋、小鸡、母鸡、鸡舍。
- 最终物品交付。
- 1 个建筑：鸡舍。

### 物流

- 货车容量。
- 简单补货规则。
- 离线补货。
- 棋盘满时不可拆货。
- 掉落表。
- 货车必须表现为“今天送来的东西”，不是体力条。

### 村庄

- 村庄界面或村庄区域。
- 建筑状态。
- 建筑完成反馈。
- 空地变成鸡舍。
- 鸡舍完成后有明确生活反馈。
- 鸡舍完成后村庄要比之前更有生命感。

### UI

- 货车货量。
- 当前目标。
- 拆货按钮。
- 村庄入口。
- 建筑需求显示。
- 基础设置。

UI 只做必要展示，不做复杂任务面板。

## 建议 MVP 内容

Merge 链：

```text
鸡蛋 -> 小鸡 -> 母鸡 -> 鸡舍
```

建筑：

- 鸡舍。

资源：

- 使用统一占位 PNG。
- 所有资源通过 `AssetManager` 访问。
- 正式资源后续替换，不影响玩法验证。

## 当前实现范围

- 货车会按时间补货，初始提供足够货物用于 10 分钟验证。
- 拆货会把鸡蛋放入第一个空格。
- 棋盘固定 6x6。
- Merge 链只包含鸡蛋、小鸡、母鸡、鸡舍。
- 鸡舍生成后会飞入村庄，不继续留在棋盘。
- 村庄空地会切换为鸡舍状态。
- 当前生活反馈是文字和村庄状态变化，尚未做人、鸡和声音等正式生活表现。

## 坚决不做

- 不做无限金币挂机。
- 不做复杂订单系统。
- 不做大量建筑。
- 不做第二条 Merge 链。
- 不做鸡舍后的延伸链。
- 不做装饰装修系统。
- 不做多人社交。
- 不做排行榜。
- 不做复杂活动。
- 不做付费系统。
- 不做多地图。
- 不做棋盘扩建。
- 不做多格物品。
- 不做复杂生产器。
- 不做自动 Merge 作为默认能力。
- 不做每日签到。
- 不做大额离线收益结算。
- 不做让村民变成订单板的系统。

## MVP Review 问题

完成后必须回答：

- 拆货是否有爽感？
- 6x6 是否合适？
- 玩家是否理解最终物品进入村庄？
- 鸡舍完成后，玩家是否能说出村里发生了什么？
- 村庄变化是否比 Merge 本身更让人记住？
- 玩家是否有理由第二天回来？
- 棋盘堵死是否可控？
- 货车补货节奏是否合适？
- 占位资源是否足够支撑后续正式美术替换？

如果答案不明确，优先调核心循环，不扩内容。
