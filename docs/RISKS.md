# RISKS

本文件主动分析 Village Merge 可能失败的地方，并提出删改建议。

## 最大风险：村庄变化不够强

如果建筑建成后只是多一个图标，项目会立刻变成普通 Merge。

风险表现：

- 玩家主要停留在棋盘。
- 建筑只是任务终点。
- 村庄没有人物、声音、行为变化。
- 玩家记不住今天村里发生了什么。

优化建议：

- MVP 必须优先验证“建成鸡舍后村庄真的变活”。
- 鸡舍完成后至少要有 3 个生活反馈：视觉、行为、声音或对话。
- 每个建筑必须先写“生活变化清单”，再写资源功能。

删除建议：

- 删除只提供数值加成的建筑。
- 删除不能在村庄中被看见或感受到的升级。

## 风险：变成普通 Merge 游戏

最容易滑向：

- 大量物品链。
- 大量订单。
- 高级物品反复消耗。
- 棋盘越来越复杂。
- 玩家只追下一级图标。

这条路成熟，但竞争极强。

优化建议：

- 限制早期物品链数量。
- 每条链必须有村庄意义。
- 合成终点要进入村庄，而不是留在棋盘。
- 不用“订单板”作为主要驱动。

删除建议：

- 删除纯粹为了填内容的链。
- 删除无法改变村庄生活的高级物品。

## 风险：变成普通经营游戏

如果建筑开始围绕生产、加工、销售、扩张展开，游戏会变成轻经营换皮。

风险表现：

- 鸡舍产鸡蛋。
- 磨坊产面粉。
- 市场卖商品。
- 玩家优化产线。
- 村民成为订单来源。

这些内容并非绝对不能做，但不能成为核心。

优化建议：

- 建筑功能先服务生活，再服务经济。
- 市场应首先是村里变热闹的地点，其次才是交易功能。
- 产出必须有限、轻量、服务事件，而非形成无限循环。

删除建议：

- 删除无限产出建筑。
- 删除多建筑乘法加成。
- 删除需要玩家频繁收取的生产计时。

## 风险：变成普通放置游戏

货车补货很容易被玩家理解为体力或离线收益。

风险表现：

- 玩家离线越久，只是货越多。
- 上线先清收益。
- 系统鼓励定时回来收。
- 村庄变化依赖等待，而非行动。

优化建议：

- 货车应该带来“今天的事情”，不只是货物数量。
- 离线补货要有限，避免收菜感。
- 每次上线要先给村庄观察点，而不是先弹奖励。

删除建议：

- 删除大额离线收益结算。
- 删除持续产币建筑。
- 删除强迫定时上线的冷却设计。

## 风险：返乡大学生身份只是文案

如果玩家身份不影响系统，设定会变成表皮。

风险表现：

- 玩家行为和村长、农场主没有区别。
- 物流和互联网只是名词。
- 村民没有人与人的关系。

优化建议：

- 所有关键系统都要能解释为返乡大学生的行动。
- 物流代表连接外部资源。
- 互联网代表帮助村庄被看见。
- 活动和游客代表村庄重新与外部发生关系。

删除建议：

- 删除“上帝视角建造一切”的表达。
- 删除让玩家像企业主一样压榨产出的系统。

## 风险：治愈感和商业化冲突

微信小游戏需要商业化，但 Village Merge 的核心情绪怕焦虑。

风险表现：

- 红点过多。
- 限时礼包压迫。
- 不上线损失。
- 体力不足阻断。
- 大量倒计时。

优化建议：

- 商业化应围绕便利、外观、活动支持，不围绕焦虑。
- 付费不能破坏“村庄慢慢变好”的节奏。
- 广告和奖励要包装成物流帮助或外部支持，但不能喧宾夺主。

删除建议：

- 删除惩罚性离线损失。
- 删除强制广告复活式设计。
- 删除破坏宁静感的商业弹窗。

## 风险：10 分钟内没有满足感

如果建设目标过大，玩家每天只是在攒材料，会流失。

风险表现：

- 今天无法完成任何可见变化。
- 玩家只获得材料。
- 任务进度条增长很少。
- 村庄没有反馈。

优化建议：

- 大建筑拆成可见阶段。
- 每天至少有一个小生活反馈。
- 即使建筑未完成，也要有“清理、搭框架、有人期待”的状态变化。

删除建议：

- 删除第一天无法完成的超长目标。
- 删除只靠进度条表达建设的任务。

## 风险：故事太重，不适合微信小游戏

返乡设定很有潜力，但不能做成重剧情。

风险表现：

- 长对白。
- 强制剧情。
- 复杂人物关系。
- 玩家想玩却被文字打断。

优化建议：

- 用场景变化讲故事。
- 用短句对话。
- 用人物行为替代长文本。
- 每次故事只提供一个情绪点。

删除建议：

- 删除长篇剧情章节。
- 删除开场大量背景设定。

## 风险：村庄生活成本过高

让村庄“活起来”比做数值系统更贵。

风险表现：

- 每个建筑都需要大量动画和事件。
- 内容生产跟不上。
- 系统变复杂但玩家感知弱。

优化建议：

- 建立可复用生活反馈模板。
- 每个建筑只要求少量高价值反馈。
- 优先做“可复用行为”：走路、停留、喂食、围观、聊天、叫声。
- 用轻量随机事件制造变化。

删除建议：

- 删除每个建筑都要定制大量复杂动画的要求。
- 删除玩家看不见的后台生活模拟。

## 当前最需要挑战的设定

### “建筑改变玩法”听起来好，但容易过度系统化

如果每个建筑都解锁新玩法，后期复杂度会爆炸。

替代方案：

- 建筑优先改变村庄生活。
- 少数关键建筑改变玩法。
- 多数建筑改变事件池、人物行为和场景状态。

### “物流、互联网、活动、游客”都合理，但不能同时早做

这些方向都成立，但早期全做会没有重点。

替代方案：

- MVP 只做物流和一个建筑。
- 第二阶段只加入一个村庄事件。
- 互联网和游客等到村庄有足够内容后再出现。

### “每天发生新事情”内容压力很高

如果理解为每天都要全新内容，团队会撑不住。

替代方案：

- 做可组合事件池。
- 用建筑状态、人物、天气、货车内容组合出变化。
- 让玩家感到新鲜，不要求每天手工写新剧情。

## 最重要的 MVP 验证

MVP 不应该验证复杂系统。

MVP 应验证：

```text
玩家完成鸡舍后，是否因为村里出现了新的生活而感到满足，并愿意明天回来。
```

如果这个不成立，继续加链、加建筑、加经济都没有意义。
