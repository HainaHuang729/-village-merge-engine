# DECISIONS

本文件记录关键决策。每条决策应说明背景、选择和影响。

## D-001：先文档，不先写代码

日期：2026-06-25

决策：第一阶段只创建项目文档，不开始游戏实现。

原因：

- 当前项目仍是 quickstart 示例。
- Village Merge 需要先明确核心循环和系统边界。
- 过早写代码容易把示例结构延续成不可维护架构。

影响：

- 暂不修改 `game.js` 和 `js/` 游戏逻辑。
- 后续先 Review 文档，再进入 MVP 框架开发。

## D-002：棋盘默认固定 6x6

日期：2026-06-25

决策：MVP 使用固定 6x6 棋盘。

原因：

- 适合微信小游戏竖屏触控。
- 能产生整理压力。
- 控制复杂度。

影响：

- 第一版不做棋盘扩建。
- 第一版不做多格物品和障碍格。

## D-003：建筑一次性完成，不重复刷同一建筑

日期：2026-06-25

决策：建筑作为永久村庄成长节点，默认只完成一次。

原因：

- 避免重复劳动。
- 强化村庄真实成长。
- 防止变成无限交付订单。

影响：

- Merge 链需要在建筑完成后延伸或分支。
- 建筑不能只做数值奖励。

## D-004：经济不以无限金币为核心

日期：2026-06-25

决策：MVP 不把金币作为核心进度资源。

原因：

- 无限金币容易引发经济膨胀。
- 项目的核心是拆货、整理、Merge 和建设。
- 建筑应改变规则，而不是单纯增加产出。

影响：

- 资源重点放在货物、棋盘空间、建筑状态和时间。
- 商业化设计后置。

## D-005：物流包装替代体力包装

日期：2026-06-25

决策：不使用“体力”作为核心概念，使用货车、配送、货位、物流表达行动限制。

原因：

- 贴合世界观。
- 玩家回来处理积压货物更自然。
- 比传统体力更符合“拆货整理”的体验。

影响：

- UI 显示货车容量和下一件配送时间。
- 行动限制来自货车和棋盘空间。

## D-006：PixiJS 作为默认技术方向（已废弃）

日期：2026-06-25

决策：文档阶段曾以 TypeScript + PixiJS 作为默认技术方向，Phaser 保留为备选。

原因：

- 核心玩法是自定义 2D 棋盘和 UI。
- 不依赖复杂物理和关卡编辑器。
- PixiJS 更适合做轻量渲染层和平台适配。

影响：

- MVP 框架设计会按渲染层、系统层、平台层分离。
- 最终框架选择仍需在开发前确认。

状态：已被 D-010 取代。当前 Phase 1 使用 Phaser 3。

## D-007：第一版只保留鸡舍链

日期：2026-06-25

决策：MVP 只保留一条 Merge 链：鸡蛋、小鸡、母鸡、鸡舍。

原因：

- 先验证拆货、整理、Merge、建筑入村的基础闭环。
- 避免过早扩展内容导致资源、数值和系统复杂度上升。
- 美术资源可以先围绕一条链建立统一规范。

影响：

- 磨坊、面包房、蜂场、果园等全部后置。
- 鸡舍后的冠军鸡、金鸡蛋也后置。

## D-008：所有资源通过 AssetManager 访问

日期：2026-06-25

决策：建立资源 manifest 和 `AssetManager`，业务代码后续只使用 Asset ID，不硬编码图片路径。

原因：

- 占位资源未来需要批量替换正式美术。
- 微信小游戏和 Web 的路径适配要集中处理。
- 预加载、缺失兜底和版本管理需要统一入口。

影响：

- 新增资源必须先登记到 Game 或 Engine 对应的 asset manifest，并通过 `packages/merge-engine` 的 `AssetManager` 加载。
- 游戏代码不得直接写 `assets/...` 路径。

## D-009：项目转为 Engine + Game 双层架构

日期：2026-06-25

决策：将项目主线从单一 Village Merge 转为 Reusable Merge Engine。目录采用 `packages/merge-engine` 和 `games/village-merge`。

原因：

- Merge 核心能力应可复用到多个题材。
- Village Merge 只是第一个 Demo。
- 防止具体游戏逻辑污染通用引擎。

影响：

- Engine 不允许包含具体题材代码。
- Game 负责物品链、经济、UI、事件和存档结构。
- 后续所有通用能力优先进入 Engine，题材能力留在 Game。

## D-010：使用 Phaser 3 而不是 Phaser 4

日期：2026-06-25

决策：按当前目标使用 Phaser 3，并在依赖中固定 `phaser@3.90.0`。

原因：

- 用户明确要求 Phaser 3。
- Phaser 4 API 仍可能带来迁移不确定性。
- Phaser 3 对 Vite + TypeScript 的项目结构足够稳定。

影响：

- Engine peer dependency 限制为 `>=3.80.0 <4`。
- 后续如升级 Phaser 4，必须单独评估。

## D-011：文档权威顺序以 North Star 和 Engine/Game 边界为准

日期：2026-06-25

决策：后续所有设计争议优先参考 `NORTH_STAR.md`、`GAME_VISION.md`、`GAME_PILLARS.md`、`RISKS.md`、`ENGINE_ARCHITECTURE.md`、`SYSTEM_DESIGN.md` 和 `MVP.md`。

原因：

- 项目经历了从 Village-only 到 Reusable Merge Engine 的方向调整。
- 早期文档中存在 PixiJS、更多链路、更多建筑和金币占位等旧表达。
- 需要一个明确阅读顺序，防止后续开发把旧设定当成当前目标。

影响：

- 如果旧文档和权威文档冲突，以权威文档为准。
- Engine 不允许具体题材代码。
- Village Merge MVP 只验证鸡舍闭环和村庄生活反馈。
- 金币相关资源命名保留为历史占位，不代表核心经济方向。
